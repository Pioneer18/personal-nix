# Don't shadow OAuth subscription with ANTHROPIC_API_KEY

**Status**: Accepted — 2026-05-16.

**Scope**: Pioneer18's MacBook-Pro-2. Applies to every shell sourcing `~/.secrets` and every spawned `claude -p` / `claude` invocation.

## Context

Pioneer18 holds two Anthropic credentials:

1. **Personal Max subscription** — OAuth token in keychain (`claude_code_oauth`), exported as `CLAUDE_CODE_OAUTH_TOKEN`. This is what should pay for daily `claude` / `claude -p` usage.
2. **RelyMD API key** — long-lived API key in keychain (`anthropic_api_key`). Used by specific apps that talk to the Anthropic SDK directly (e.g. `mcps/tachikoma-ui/server`). Billed to RelyMD.

The `claude` CLI auth precedence is: `ANTHROPIC_API_KEY` (env) > OAuth keychain. When both are set in shell env, claude silently bills the API key — *not* the subscription.

Until 2026-05-16, `~/.secrets` (generated by `scripts/secrets-from-keychain.sh`) exported the RelyMD key as the bare `ANTHROPIC_API_KEY`. Every interactive shell, MCP server, and tachikoma supervisor inherited it. Two failure modes resulted:

- **Silent mis-billing**: claude sessions ran on RelyMD's API quota when Pioneer18 intended subscription usage.
- **Invalid-key crashes**: when the RelyMD key expired/rotated, spawned `claude -p` sessions died at startup with `Invalid API key`, even though the user's subscription was fully valid. The 4 in-flight tachikomas were unaffected because [`skills/tachikoma/tachikoma.sh.tmpl`](../../skills/tachikoma/tachikoma.sh.tmpl) already does `env -u ANTHROPIC_API_KEY` before invoking `claude -p` — a defensive workaround pattern also documented in [`healix-insight-lab-cli-over-api.md`](./healix-insight-lab-cli-over-api.md). But ad-hoc `nohup claude -p ...` launches and one-off scripts did not have the workaround, and hit the failure.

The defensive pattern works but is easy to forget. The real fix is to stop shadowing the OAuth credential in the first place.

## Decision

**`ANTHROPIC_API_KEY` is never set in `~/.secrets` or shell env.** The RelyMD key is exported as **`ANTHROPIC_API_KEY_COMPANY`** instead.

- The keychain entry name (`anthropic_api_key`) is unchanged — that's the persistent secret storage.
- `scripts/secrets-from-keychain.sh` exports the value under the `_COMPANY`-suffixed env var.
- Apps that need to use the RelyMD key explicitly read `ANTHROPIC_API_KEY_COMPANY` (e.g. `mcps/tachikoma-ui/server/{claude,index}.ts`).
- `claude` CLI sees no `ANTHROPIC_API_KEY` in env → falls back to OAuth keychain → bills the Max subscription. Default-correct.

The `env -u ANTHROPIC_API_KEY` defensive pattern in `tachikoma.sh.tmpl` stays as belt-and-suspenders, but is no longer load-bearing.

## Consequences

**Good**:
- Default-correct billing. New shells, fresh `claude -p` invocations, ad-hoc skill launches all use the subscription. No env-unsetting required.
- Stale/expired RelyMD keys can't silently break claude sessions.
- The intent of each credential is encoded in its env var name — `_COMPANY` for RelyMD-billed, plain `CLAUDE_CODE_OAUTH_TOKEN` for personal subscription.

**Trade-offs**:
- **Third-party tools that read `ANTHROPIC_API_KEY` automatically (Python SDK, generic MCP servers, ad-hoc scripts) no longer find a key.** Those need either: (a) an explicit re-mapping (`ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY_COMPANY some-tool`), or (b) a code update to read the new var. This is an intentional friction — forcing every API-key-billed use to be explicit so subscription-billed use is the default.
- **Existing shells still have the old `ANTHROPIC_API_KEY` until reopened.** `unset ANTHROPIC_API_KEY` in active sessions, or open new terminals.

## Migration

1. Updated `scripts/secrets-from-keychain.sh` line 39: export name changed.
2. Updated `mcps/tachikoma-ui/server/claude.ts` and `index.ts` to read `ANTHROPIC_API_KEY_COMPANY`.
3. Regenerated `~/.secrets` via `scripts/secrets-from-keychain.sh`.
4. **User action required in already-open shells**: `unset ANTHROPIC_API_KEY && source ~/.secrets`.
